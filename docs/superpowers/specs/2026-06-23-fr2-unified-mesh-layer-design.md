# FR-2: Unify the mesh layer (one bitmask-driven composite edge tile)

**Date:** 2026-06-23
**Status:** Design — part 2 of 3 in the field-driven terrain rewrite
(FR-1 unify descriptor + water → **FR-2 unify mesh layer** → FR-3 field-driven
decoration scatter + delete socket engine).

## Why

This is the highest-leverage simplification. The largest body of special-cased
content is the per-variant scene system: **~14 authored sheer-cliff scenes
(`terrain/scenes/cliff/`, reused by banks), ~15 authored level scenes
(`terrain/scenes/level/`), and ~35 generated slope scenes
(`terrain/scenes/slope/`)** — plus the parallel `*_VARIANT_TABLE`s and
`SlopeVariantLayout.VARIANT_MASKS`. They all encode the same thing: a tile's edge
shape, given which neighbours are lower.

The project has already proven the escape route. `SlopeMeshGenerator` +
`SlopeProfile` **generate** the sloped cliff meshes at runtime/bake from four
canonical height functions (`edge`, `outer_corner`, `inner_corner`, + two stacked
corner profiles), and `SlopeVariantLayout.layout()` composes any variant as an
**N×N grid of those components, placed by a bitmask with rotation**. That is
exactly the uniform scheme we want — it is simply not applied to level, cliff
(sheer), or bank yet.

FR-2 extends that one component-composition model to every edge family,
parameterized by **drop height** and **top material**, and deletes the authored
variant scenes, the four variant tables, and `VARIANT_MASKS`.

## Goal

A single `composite_edge_tile(edge_bitmask, drop_height, material) → mesh/scene`:
- The N×N (N=2) grid of canonical components `{top, edge, outer_corner,
  inner_corner, *_stacked}` chosen per cell **directly from the FR-1
  `edge_bitmask`** (deriving the layout from the bitmask, not from a per-variant
  `VARIANT_MASKS` table), rotated by the existing `EDGE_ANGLE`/`CORNER_ANGLE`.
- The component meshes generated by an extended `SlopeMeshGenerator` /
  `SlopeProfile` parameterized by `drop_height` (≈0.5 for a level step, ≈4 for a
  cliff/bank storey) and `top_material` (grass for land tops, water-floor/rock for
  bank walls).
- The 2-storey diagonal-corner ramp (`outer_corner_stacked`/`inner_corner_stacked`,
  the `generated_stacked_variants` subsets) generalized to a **corner-depth**
  parameter from FR-1's `corner_depths`, not a per-subset enumeration.

Result: one component set (~4–6 generated meshes per drop height) + one
`bitmask → grid` function replaces all authored level/cliff/bank scenes and all
four variant tables.

## Generation strategy: extend the bake, keep it offline

Two options were considered:
- **Runtime composition** (compose meshes on the fly per cell): most elegant, but
  adds per-placement mesh cost and complicates the streaming hot path.
- **Offline/asset-time generation** (extend `bake_slope_cliffs.gd` to emit the
  full family of variant scenes from the one component set): lower risk, matches
  the already-shipping slope path, keeps placement a cheap scene-instance.

**Decision: extend the offline bake** to generate every edge variant (level/cliff/
bank) from the parameterized component set, producing the variant scenes the
instantiator instances. This keeps runtime placement unchanged (instance a scene
by computed key) while eliminating the *authored* scenes and the hand-maintained
tables — the scenes become a build artifact of one generator, not content. (A
later move to pure-runtime composition stays open but is out of scope here.)

## What gets deleted

- The authored `terrain/scenes/level/*.tscn` (15) and `terrain/scenes/cliff/*.tscn`
  (14, the sheer scenes banks reused) — regenerated from the component set.
- `LEVEL_VARIANT_TABLE`, `CLIFF_VARIANT_TABLE`, `CLIFF_STACKED_VARIANT_TABLE`, and
  the `BANK_VARIANT_TABLE` already removed in FR-1.
- `SlopeVariantLayout.VARIANT_MASKS` — the layout is derived from the FR-1
  `edge_bitmask` directly; `generated_stacked_variants` becomes a corner-depth
  function rather than an enumerated subset list.
- Any remaining `HeightfieldVariant`/`SlopeVariantLayout` table that duplicates the
  one canonical catalog.

## What is kept / extended

- `SlopeMeshGenerator`, `SlopeProfile`, `SlopeAtlas`, `bake_slope_cliffs.gd` —
  **extended** to parameterize drop height + top material and to emit all families
  (renamed appropriately, e.g. a generic `bake_edge_tiles.gd`).
- The canonical variant catalog (one list of ~15 rotational orbits) and the
  `bitmask → (shape, rotation)` function from FR-1.
- The generated variant scenes (now a build artifact) and the instancing path in
  `HeightfieldInstantiator`.
- **Authored foliage gltf art is untouched** — grass/rock/bush/tree/hill meshes
  are art, not geometry-derivable; generating them would be over-engineering.

## The hard case: 2-storey diagonal corners

The clamp deliberately allows a corner to drop two storeys diagonally
(`HeightfieldPlan` understacks; `HeightfieldInstantiator._add_understack_corners`).
This is genuine ramp geometry (`outer_corner_stacked` / `inner_corner_stacked`,
the C1-mating `SlopeProfile._top_s`/`_bot_s` profiles). FR-2 generalizes it to a
**corner-depth parameter** (drop in storeys at that corner) feeding the component
generator — NOT a fully general N-storey ramp mechanism (that would be cleverness
that hurts readability). Keep it to "corner-drop-depth bitmask → component."

## Risks (HIGH — this is the visual/art change)

- **Appearance regression.** Level/cliff/bank meshes must look right at their
  parameterized drop heights. `SlopeProfile`'s curves are tuned for the 4m cliff;
  applying them at 0.5m (level) and at water-floor depth needs validation.
  Mitigate: generate, instance, and **screenshot every variant** side-by-side with
  the current authored scene before deleting the authored versions.
- **Collision shapes.** The generated components include collision
  (`build_*_collision`); the parameterized versions must produce correct walkable
  collision at each drop height. Guard with the slope collision tests, extended.
- **Socket/marker parity (transitional).** While FR-3 still uses sockets for deco,
  the generated variant scenes must carry the same surface markers the authored
  ones did (the bake already copies sockets via `_ground_surface_sockets`). After
  FR-3 deletes sockets, this constraint relaxes. Sequence: if FR-2 runs before
  FR-3, preserve socket parity in the generated scenes.
- **Build/bake step in the pipeline.** Variant scenes become generated artifacts;
  document the bake command and ensure tests/CI regenerate or check them.

## Testing / verification

- Extend `test_slope_*` (mesh generator, profile, variant layout, components) to
  cover the parameterized drop heights and materials; assert the generated
  level/bank variants match expected geometry (vertex/AABB/collision checks).
- A generation test: for each of the ~15 canonical shapes × {level, cliff, bank},
  the bake produces a valid scene with correct walkable-surface height and
  collision.
- Full GUT suite green.
- **Visual (primary):** regenerate all variants, run the game, screenshot terrain
  with cliffs, terraces (levels), and shorelines (banks); compare against the
  authored-scene baseline. The terrain must look the same (or cleaner at seams).

## Dependencies

Depends on FR-1 (consumes the unified `edge_bitmask` / `drop_height` / `material`
descriptor and the one canonical catalog). Order vs FR-3 is flexible; recommended
last so it lands on a field-only base, but if socket-deco (FR-3) is still live,
preserve surface-marker parity in the generated scenes.
